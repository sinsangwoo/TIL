## 📅 42일 차: I/O 디바이스 모델링 및 CPU-I/O 중첩

### 🎯 오늘의 목표
1. `IODevice` 클래스를 만들어 디스크와 같은 외부 장치를 추상화한다.
2. OS가 I/O 시간을 직접 세지 않고, 디바이스에 위임한 뒤 완료 인터럽트를 받는 구조로 리팩토링한다.
3. 한 프로세스가 I/O를 하는 동안 다른 프로세스가 CPU를 사용하는 '중첩(Overlap)' 현상을 확인한다.
4. Burst Time과 Behavior의 개념을 명확히 분리하여 프로세스 실행 모델을 정교화한다.

### 📝 배운 점 (TIL)
* **멀티프로그래밍의 본질:** CPU와 I/O 장치가 동시에 일하게 만드는 것. P1이 I/O 때문에 잠들었을 때 P2가 CPU를 차지하는 로그를 보며 멀티 프로그래밍의 본질을 느꼈다.
* **하드웨어 인터럽트 시뮬레이션:** `disk.finished_queue`를 폴링(Polling)하는 방식으로 인터럽트를 흉내 냈다. 실제 OS의 인터럽트 핸들러가 하는 역할이 바로 이거다.
* **프로세스 실행 모델의 정교화:** burst_time은 단순한 '총 시간'이 아니라 '순수 CPU 점유 시간'임을 정의했다. 이를 통해 I/O가 발생하는 시점에 제어권을 OS로 넘기는 트랩(Trap) 메커니즘을 코드 수준에서 명확히 구현했다.

### 💥 트러블 슈팅
* 문제: TypeError 발생 (Process 생성자 인자 불일치)
* 원인: Process 클래스는 burst_time을 필수 인자로 받도록 되어 있었으나, main.py에서 behavior 리스트만 전달하여 발생했다.
* 해결: burst_time에 기본값(0)을 부여하고, behavior가 입력되면 리스트 내의 CPU 작업 시간 합계를 자동으로 계산하도록 __init__ 로직을 개선했다. 이 과정에서 사용자 정의 행동(Behavior)과 시스템 메트릭(Burst Time)의 동기화가 중요하다는 것을 배웠다.

* 문제: SyntaxError (괄호 미닫힘 에러)
* 원인: __repr__ 메서드 수정 중 f-string을 감싸는 괄호를 닫지 않아 발생했다.
* 해결: 파이썬 해석기의 에러 메시지를 통해 해당 줄의 괄호 쌍을 확인하고 수정했다. 코드가 복잡해질수록 괄호 짝 맞추기와 같은 기본적인 문법 체킹의 중요성을 다시 한번 상기했다.
